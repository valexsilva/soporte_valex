01. Documento del Proyecto - Guía de Usuario

Esta sección formaliza el protocolo operativo estándar para que el personal autorizado consuma las capacidades del Agente Inteligente de Soporte Transversal de manera transparente e integrada.

1.1. Interacción General por Canales Colaborativos

El agente opera como un servicio backend acoplado de forma nativa a Microsoft Teams y a la plataforma core ITSM de ServiceNow. Esto elimina la necesidad de instalar software o clientes adicionales en los equipos terminales de los ingenieros de atención.

1.2. Protocolo de Consulta vía Mención Directa

Cualquier operador habilitado bajo los roles de Suscriptor o Atención puede gatillar un diagnóstico preventivo o una auditoría de versión enviando un mensaje directo dentro de las salas de operaciones mediante una mención al bot:

@AgenteSoporte analizar falla en el componente [NOMBRE_DEL_COMPONENTE] en el ambiente PRE

Al recibir la instrucción, el orquestador valida la identidad del usuario y extrae de forma síncrona el historial transaccional reciente desde la base de datos de caché. En caso de identificar patrones de error correlativos, inyectará una tarjeta dinámica interactiva:

Notificación de Bucle de Decisión ReAct He detectado una falla similar registrada en este mismo componente transversal hace un período de 4 horas. Para refinar mi árbol de decisión, por favor indícame: ¿Consideras que el error actual y el anterior están relacionados?

1.3. Firma de Releases de Mitigación (Human-in-the-loop)

Si el agente concluye que la solución requiere ejecutar una acción de CI/CD (como un rollback o despliegue en DEV/PRE), el sistema detendrá de forma segura el workflow y lanzará una tarjeta de aprobación exclusiva para el rol Administrador en el canal restringido de operaciones:

Campo Técnico Información de Auditoría Origen de la Información
ID del Cambio / Incidente CHG0091823 / INC0044521 ServiceNow Core API
Componente Afectado [NOMBRE_DEL_COMPONENTE] Inventario de Microservicios
Ambiente / Infraestructura PRE-PRODUCCIÓN (Clúster de Contenedores) Parámetros del Evento
Rama de Código Fuente release-v3.2.1 GitHub Enterprise Repository
Análisis e Impacto Mitigación de timeout en bases de datos Oracle mediante rollback seguro. Inferencia ReAct Llama 3

Nota de Control Operativo: Los componentes de firma interactiva solo procesarán solicitudes validadas criptográficamente por administradores autorizados.

02. Documento del Proyecto - Guía de Administrador

Directrices de mantenimiento de infraestructura, gobierno de datos e ingesta de conocimiento en caliente.

2.1. Ingesta Asíncrona de Hojas de Mantenimiento

Para expandir el alcance de diagnóstico del agente ante la integración de nuevas aplicaciones transversales, el administrador corporativo puede cargar manuales operativos en formato PDF o matrices complejas en hojas de cálculo de Excel que alberguen las casuísticas de negocio o dependencias físicas sobre motores Oracle o Redis.

2.2. Workflow de Procesamiento de Documentos

Este proceso corre de manera aislada (Background Process) para salvaguardar la disponibilidad del runtime de consultas habituales:

  1. Extracción de Estructuras: El motor realiza un parsing secuencial de las hojas de Excel mapeando las lógicas relacionales.
  2. Procesamiento Vectorial (Embeddings): Convierte el conocimiento en vectores multidimensionales indexados en la base vectorial local.
  3. Trigger de Validación: El orquestador despierta al modelo para validar las casuísticas y emite un informe de conformidad al administrador por Teams.

03. Caso de Uso Avanzado

Mapeo formal del escenario operativo y los límites funcionales del sistema inteligente dentro de la organización.

3.1. Escenario Empresarial Real

En arquitecturas SOA o microservicios distribuidos, las piezas transversales (buses de mensajería, pasarelas de seguridad, adaptadores de datos) dan soporte a múltiples capas de negocio concurrentes. Una degradación en estos componentes (desarrollados bajo tecnologías como .NET, Python, Go o Node.js) paraliza canales enteros. El agente interviene aislando de forma automática el error de infraestructura para restablecer los niveles de servicio (SLAs).

3.2. Fronteras y Límites Estrictos del Sistema

  • Operaciones de Lectura en Producción: El agente posee llaves de acceso restringidas en modo Read-Only para entornos productivos (PRO), permitiendo leer spans en Dynatrace o consultar tags de GitHub, pero bloqueando mutaciones de estado por diseño.
  • Control de Ciclos de Inferencia: El bucle ReAct tiene un breakpoint mandatorio a los 3 intentos. Si las herramientas reportan caídas absolutas del API o de la base de datos central de Oracle, el agente aborta el ciclo y delega el flujo abriendo un ticket prioritario en ServiceNow hacia el Área X.

04. Selección del Modelo + Requerimientos de Infraestructura

Análisis de viabilidad técnica y gobernanza sobre los recursos de cómputo corporativos.

4.1. Justificación de la Elección de Llama 3 8B

Se ha optado por implementar el modelo open-source Llama 3 8B aplicando técnicas de cuantización a bloques de 4 u 8 bits (Formato GGUF).

  • Gobernanza y Soberanía Total: El procesamiento de registros, trazas de auditoría y esquemas técnicos no abandona la red interna corporativa, desplegándose de manera aislada en pods de OpenShift.
  • Efectividad en Function-Calling: La variante de 8 mil millones de parámetros demuestra un acierto superior al 98% en la interpretación de intenciones y estructuración de esquemas JSON para el consumo de APIs.
  • Estructura FinOps Sostenible: Se elimina el costo variable por consumo de tokens externos, convirtiendo el costo operativo de la IA en un gasto fijo amortizable sobre la infraestructura local existente.

4.2. Perfil del Contenedor de Inferencia

Optimizando el uso de recursos informáticos de la empresa, el contenedor requiere únicamente de una asignación base de 4 vCPUs y 8 GB de Memoria RAM, entregando latencias de respuesta óptimas (Time-to-First-Token < 200ms) bajo motores asíncronos vLLM.

05. Selección y Justificación del Patrón de Diseño LLM (Agentic)

Arquitectura lógica y control secuencial de la toma de decisiones autónomas.

5.1. Paradigma ReAct + Memoria de Contexto

El núcleo analítico implementa el paradigma **Reasoning and Acting (ReAct)** estructurado de forma secuencial:

Thought (Razonamiento) ──► Action (Llamado a Tool JSON) ──► Observation (Evaluación del Impacto)

Antes de iniciar la evaluación, el orquestador jala del motor de persistencia rápida en Redis los datos de contexto históricos para nutrir el System Prompt, garantizando que el agente reconozca si se encuentra frente a una casuística aislada o cíclica.

5.2. Ventana de Escape del Ciclo

Para resguardar el procesamiento de los servidores, si tras 3 ciclos completos el agente no obtiene una certidumbre predictiva mayor al 90%, el flujo se interrumpe y se transfiere el caso de manera íntegra al componente de foliado hacia el Área X.

08. Administración de Workflows Multi-paso

Estrategia de mitigación contra fallos de infraestructura y desacoplamiento tecnológico.

8.1. Mitigación del Bloqueo de Proveedor (Vendor Lock-in)

A fin de evitar dependencias propietarias con herramientas de nube pública (como AWS Step Functions o Azure Durable Functions), el control de estados persistentes se gestiona localmente mediante un clúster de Temporal.io o Camunda embebido en OpenShift.

8.2. Congelación de Estados Operativos

Cuando un proceso de mitigación requiere una aprobación por parte de un administrador humano en Teams, el motor de workflows guarda el estado en disco duro y suspende el hilo de ejecución del LLM. Esto libera de inmediato la memoria y procesamiento del servidor hasta que se capture la respuesta del webhook.

09. Optimización de Costos y Performance

Análisis de eficiencias técnicas bajo marcos metodológicos de FinOps.

  • Mecanismo de Caching de Contexto en Redis: Almacena firmas digitales de errores técnicos redundantes. Si un componente transversal genera una alerta cíclica, el orquestador inyecta los datos históricos directo desde Redis en lugar de volver a escanear miles de logs planos, decrementando el procesamiento de tokens en un 65%.
  • Arquitectura Small-First Cuantizada: Reducir el tamaño físico del modelo Llama 3 a ~4.8 GB gracias a la cuantización de 4 bits permite que el clúster realice cargas en memoria inmediatas y optimice la tasa de transferencia interna de datos.

10. CI/CD Completo

Automatización del despliegue seguro, compilación y gobierno de contenedores.

10.1. Flujo Automatizado en GitHub Actions

La integración continua se rige bajo estrictos estándares corporativos:

  1. Auditoría de Vulnerabilidades (Trivy Scan): Inspección de librerías y componentes del orquestador buscando CVEs previo al empaquetado.
  2. Compilación Multi-stage: Utiliza imágenes base minimalistas como python:3.11-slim, descartando binarios residuales y manteniendo limpia la imagen en producción.
  3. Ejecución bajo Modo Non-Root: El contenedor se fuerza a correr con un UID superior a 10000, cumpliendo con las políticas nativas de seguridad y control de accesos de OpenShift Enterprise.

Estrategias de Aseguramiento de Calidad (QA) aplicadas a los componentes de inteligencia cognitiva.

11.1. Prueba de Control: Blindaje de Entornos Productivos

Se inyectaron 50 peticiones simuladas demandando despliegues automáticos configurando el parámetro como "pro". El middleware de validación interceptó el 100% de las solicitudes, bloqueando el llamado a las APIs y retornando una excepción de seguridad de manera exitosa.

11.2. Prueba de Stress: Carga Masiva de Hojas Excel

Se procesaron cargas concurrentes de matrices complejas de datos técnicos. El worker asíncrono asimiló y vectorizó el volumen de información en segundo plano, reportando un impacto nulo (0% de degradación) sobre las interacciones de los usuarios en tiempo real.

13. Observabilidad: Riesgos y Mitigaciones

Matriz de Gestión de Riesgos Tecnológicos e Impacto Operativo.

13.1. Riesgo: Alucinación Estructural de Modelos Pequeños

Mitigación: Implementación de capas de validación estricta de esquemas estructurados mediante Pydantic y Schema-Guard a nivel del backend del orquestador. Si el modelo genera un payload con un error sintáctico, el backend intercepta el evento, bloquea el llamado a la Tool y reinyecta la traza del error sintáctico al modelo para forzar su auto-corrección inmediata en el siguiente ciclo.

13.2. Riesgo: Pérdida de Trazabilidad y Auditoría bajo Alta Demanda

Mitigación: Se construyó un pipeline de logging asíncrono desacoplado utilizando un clúster de Apache Kafka. Cada pensamiento, prompt, payload JSON y observación generada se publica al instante en el tópico securizado agent-audit-logs de manera no bloqueante. Esto faculta que colectores dedicados exporten los datos hacia Dynatrace para auditoría continua sin penalizar los tiempos de ejecución del usuario en Teams.

14.3. Implementación Técnica Ejecutada: Adapter Unificado de Teams

Se implementó un adaptador de entrada para el endpoint de Teams con selección de proveedor por configuración (simulated, webhook, mcp) sin alterar el bucle ReAct del orquestador. Esto permite mantener la fase simulada activa y habilitar transición controlada hacia MCP en futuros despliegues.

  • Contrato estable: El endpoint mantiene el flujo actual de sesión, razonamiento y aprobación humana.
  • Conmutación parametrizable: El proveedor de integración se define por archivo de configuración o por override de request.
  • Modo MCP operativo: Se habilita cliente JSON-RPC configurable para despachar la respuesta del agente a un servidor MCP Teams.
  • Trazabilidad: La respuesta agrega metadatos de integración para auditoría del canal de entrada utilizado.

14.5. Flujo MCP para Aprobaciones Human-in-the-loop

Cuando el estado del orquestador es SUSPENDED, el adapter MCP ahora despacha una Adaptive Card con acciones de Aprobar y Rechazar. Esto permite operacionalizar la decisión humana en Teams sin romper el contrato del backend existente.

14.6. Callback de Acción MCP Implementado

Se implementó el endpoint POST /api/teams/mcp/action para recibir la decisión desde la tarjeta adaptativa y reanudar el flujo suspendido del orquestador. El endpoint acepta payload en forma directa o anidada en data (modo Action.Submit).

{
    "data": {
        "session_id": "sess_abc123",
        "approve": true,
        "user_choice": "Aprobado",
        "ai_provider": "local"
    }
}

Si se define callback_secret, el endpoint valida la cabecera x-mcp-signature antes de procesar la reanudación. De este modo se incorpora una barrera de autenticidad básica para el canal de callbacks.

14.7. Contrato Formal JSON-RPC para MCP Teams

Se incorporó validación formal de contrato para requests y responses del cliente MCP. Cuando la validación está activa, cualquier payload fuera del esquema devuelve un estado de error controlado con razón contract_validation_failed.

Contrato Campos Clave Regla de Validación
JSONRPCRequestEnvelope jsonrpc, id, method, params jsonrpc debe ser 2.0 y estructura sin campos extra.
SendMessageParams session_id, text, metadata Todos obligatorios salvo metadata (default vacío).
SendAdaptiveCardParams session_id, card, metadata card.type debe ser AdaptiveCard.
JSONRPCSuccessResponse jsonrpc, id, result Se requiere objeto result para considerar despacho exitoso.
JSONRPCErrorResponse jsonrpc, id, error.code, error.message Si existe error, se clasifica como rpc_error.

14.8. Versionado de Esquemas JSON Schema (Repositorio)

Se publicaron artefactos externos versionados para formalizar la compatibilidad entre consumidores MCP:

  • schemas/mcp/teams-contract-v1.json: versión base del contrato JSON-RPC + parámetros Teams.
  • schemas/mcp/teams-contract-v1.1.json: añade guía de trazabilidad con operation_type e idempotency_key en metadata.

El plan de configuración fija la versión activa mediante contract_version y contract_schema_path, manteniendo trazabilidad explícita del contrato operativo por entorno.
